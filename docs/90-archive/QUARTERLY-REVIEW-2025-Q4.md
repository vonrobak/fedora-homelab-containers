## 2. PROBLEM ANALYSIS

### 2.1 Journal Fragmentation

**Problem:** Journals scattered across 5 directories make chronological navigation difficult.

**User pain points:**
- "How did the project evolve over time?" → Need to manually merge 5 timelines
- "What happened in November?" → Search 5 directories
- "What was the sequence of events?" → Mental timeline reconstruction required

**Example scenario:**
```
Nov 8-11: Immich deployment span
├─ 10-services/journal/2025-11-08-immich-network-and-storage-planning.md
├─ 10-services/journal/2025-11-08-week1-completion-summary.md
├─ 10-services/journal/2025-11-08-week2-day1-database-deployment.md
├─ 10-services/journal/2025-11-08-week2-day2-immich-jellyfin-optimization.md
└─ 30-security/journal/2025-11-11-authelia-deployment.md
```

User must search 2 directories to understand 1 week's work.

**Current documentation says:**
> "Understanding How We Got Here: Browse `journal/` subdirectories chronologically"

But this is impractical when entries are fragmented.

### 2.2 Inconsistent Report Naming

**Problem:** Multiple naming conventions for similar document types

**Patterns observed:**

| Pattern | Example | Count | Follows Standard? |
|---------|---------|-------|-------------------|
| `YYYY-MM-DD-desc.md` | `2025-11-09-strategic-assessment.md` | 45 | ✅ Yes (CONTRIBUTING.md) |
| `YYYYMMDD-desc.md` | `20251107-backup-implementation-summary.md` | 9 | ❌ No (legacy) |
| `UPPERCASE-NAME.md` | `PROJECT-A-DISASTER-RECOVERY-PLAN.md` | 15 | ❌ Undefined |

**Impact:**
- Sorting inconsistency (with/without hyphens sort differently)
- Unclear purpose distinctions (why UPPERCASE?)
- Harder to write tooling (multiple patterns to match)

### 2.3 Unclear Separation: Journals vs Reports

**Current definition (CONTRIBUTING.md):**
- **Journals:** "Chronological record of learning, changes, and progress" (located in subdirs)
- **Reports:** "Point-in-time system state snapshots" (located in 99-reports)

**Problem:** Many files in 99-reports are actually journals:
- `2025-11-09-strategic-assessment.md` - This is a strategic reflection, not system state
- `2025-11-12-session-summary.md` - Session work log, not state snapshot
- `2025-11-11-cli-work-session-summary.md` - Work session log
- `2025-11-14-session-3-completion-summary.md` - Session completion log

These are chronological progress logs (journals), not state snapshots (reports).

**Real reports (actual system state):**
- `SYSTEM-STATE-2025-11-06.md` - Actual infrastructure snapshot
- `20251025-storage-architecture-authoritative-rev2.md` - Storage configuration state
- `intel-*.json` - System health state
- `resource-forecast-*.json` - Predictive state

### 2.4 JSON vs Markdown Confusion

**Problem:** 99-reports contains both machine-generated JSON (94) and human-written markdown (69)

**Current state:**
- **JSON (automated):** System metrics, never edited, consumed by scripts
- **Markdown (manual):** Strategic docs, session logs, planning documents

**These serve different purposes but share the same directory.**

## 3. PROPOSED IMPROVEMENTS

### 3.1 Create /docs/98-journals (Primary Recommendation)

**Rationale:**
- Journals are fundamentally chronological, not topical
- Single location enables easy temporal navigation
- Numbering (98) places it adjacent to reports (99) - related but distinct

**Structure:**
```
docs/98-journals/
├── 2025-10/
│   ├── 2025-10-20-day01-foundation-learnings.md
│   ├── 2025-10-21-day02-networking-exploration.md
│   ├── 2025-10-23-day04-jellyfin-deployment.md
│   └── 2025-10-30-yubikey-success-summary.md
├── 2025-11/
│   ├── 2025-11-08-immich-network-and-storage-planning.md
│   ├── 2025-11-10-force-multiplier-week-days-1-5-summary.md
│   └── 2025-11-13-homelab-deployment-skill-implementation-plan.md
└── 2025-12/
    └── (future entries)
```

**Migration:**
- Move all `*/journal/*.md` → `98-journals/YYYY-MM/`
- Remove now-empty `*/journal/` subdirectories
- Update README.md and CONTRIBUTING.md to reflect new structure

**Benefits:**
1. ✅ Chronological browsing: `ls -1 98-journals/2025-11/` shows November's journey
2. ✅ Clear timeline: One directory = complete project history
3. ✅ Easier navigation: Month-based subdirs keep listings manageable
4. ✅ Preserved metadata: Files unchanged, just relocated

**Trade-offs:**
- ❌ Lose topical grouping (can't see "all security journals")
- ✅ Mitigation: Use grep/search for topical queries
- ✅ Observation: Users rarely ask "show me all security journals" - they ask "what happened in November?" or "how did we get here?"

### 3.2 Clarify /docs/99-reports Purpose

**New definition:** Machine-generated system state reports and formal snapshots

**Contents:**
1. **Automated JSON reports** (keep all)
   - `intel-*.json` - System health snapshots
   - `resource-forecast-*.json` - Predictive analytics
   - `weekly-latest.json` - Weekly rollup

2. **Formal system state snapshots** (keep)
   - `SYSTEM-STATE-YYYY-MM-DD.md` - Infrastructure state
   - Storage/architecture authoritative documents

**Move to 98-journals:**
- All session summaries (`2025-11-12-session-summary.md`)
- Strategic assessments (`2025-11-09-strategic-assessment.md`)
- Implementation summaries (`2025-11-07-backup-implementation-summary.md`)
- Session work logs (all `*-session-*` files)

**Move to appropriate guides (or archive):**
- Planning documents → Update relevant guides or archive if implemented
- Project proposals → Update guides with outcomes or archive

### 3.3 Standardize Report Naming

**JSON Reports (automated) - DO NOT CHANGE:**
```bash
# Current naming - KEEP AS-IS (automation depends on it)
intel-YYYYMMDD-HHMMSS.json
resource-forecast-YYYYMMDD-HHMMSS.json
weekly-latest.json  # Symlink to latest weekly report
```

**Rationale:** Automation dependencies require glob pattern `intel-*.json`

**Markdown Reports (manual) - STANDARDIZE:**

| Document Type | Pattern | Example |
|---------------|---------|---------|
| System state snapshot | `SYSTEM-STATE-YYYY-MM-DD.md` | `SYSTEM-STATE-2025-11-06.md` |
| Architecture snapshot | `ARCH-<subsystem>-YYYY-MM-DD.md` | `ARCH-storage-2025-10-25.md` |

**Deprecated patterns (migrate away):**
- `YYYYMMDD-description.md` → Use `YYYY-MM-DD-description.md`
- `UPPERCASE-SPECIAL-NAME.md` → Use standard `YYYY-MM-DD-description.md` in 98-journals

## 4. DETAILED MIGRATION PLAN

### Phase 1: Journal Consolidation (Low Risk)

**Objective:** Move all journals to 98-journals without breaking anything

**Prerequisites:**
- ✅ No automation reads journal files
- ✅ No hard-coded paths in scripts (verified)
- ✅ Only human navigation affected

**Steps:**

```bash
# 1. Create new structure
mkdir -p docs/98-journals/{2025-05,2025-10,2025-11}

# 2. Move journals by date (preserving metadata)
# From 00-foundation/journal/
git mv docs/00-foundation/journal/20250526-*.md docs/98-journals/2025-05/
git mv docs/00-foundation/journal/2025-10-*.md docs/98-journals/2025-10/
git mv docs/00-foundation/journal/20251026-*.md docs/98-journals/2025-10/

# 3. Repeat for all subdirectories
# (10-services, 20-operations, 30-security, 40-monitoring)

# 4. Remove empty journal directories
git rm -r docs/*/journal/

# 5. Verify no broken links
grep -r "journal/" docs/ --include="*.md"
```

**Validation:**
- [ ] All dated files successfully moved
- [ ] No journal/ directories remain (except in archive if any)
- [ ] Git history preserved (using git mv)
- [ ] No broken internal links

**Rollback:** `git reset --hard` (before committing)

### Phase 2: Report Reorganization (Medium Risk)

**Objective:** Move session logs from 99-reports to 98-journals

**Risk:** User might expect certain files in 99-reports

**Criteria for moving to 98-journals:**
1. File describes what was done (session log, implementation summary)
2. File describes strategic thinking/planning
3. File is NOT a system state snapshot

**Criteria for keeping in 99-reports:**
1. Automated JSON output
2. System state snapshot (SYSTEM-STATE-*, ARCH-*)
3. Deployment completion summaries (reference docs)

**Steps:**

```bash
# 1. Identify session logs and strategic docs
cd docs/99-reports

# Session logs (move to journals)
files_to_move=(
  "2025-11-*-session-*.md"
  "2025-11-*-strategic-*.md"
  "2025-11-*-cli-work-*.md"
  # ... (see full list in appendix)
)

# 2. Move to appropriate month
for file in "${files_to_move[@]}"; do
  month=$(echo "$file" | cut -d- -f1,2)
  git mv "$file" "../98-journals/${month}/"
done

# 3. Verify reports directory now contains only true reports
ls -1 *.md | wc -l  # Should be ~20-30 (state snapshots only)
```

**Validation:**
- [ ] 99-reports contains only system state + JSON
- [ ] Session logs in 98-journals/YYYY-MM/
- [ ] README updated to reflect changes

**Rollback:** `git reset --hard`

### Phase 3: Naming Standardization (Low Risk)

**Objective:** Standardize markdown report naming

**Old format → New format:**
```bash
# In 99-reports/ (state snapshots only)
20251025-storage-architecture-authoritative-rev2.md
  → ARCH-storage-2025-10-25-rev2.md

20251106-monitoring-stack-deployment-summary.md
  → (Move to journals - deployment log, not state)

# In 98-journals/ (session logs moved from reports)
All stay as YYYY-MM-DD-description.md (already correct)
```

**Steps:**
```bash
# 1. Rename state snapshots in 99-reports
git mv docs/99-reports/20251025-storage-architecture-authoritative-rev2.md \
       docs/99-reports/ARCH-storage-2025-10-25-rev2.md

# 2. Update any internal references
grep -r "20251025-storage-architecture" docs/ --include="*.md"
# (Manual update of links)
```

**Validation:**
- [ ] All reports follow standard naming
- [ ] No broken links
- [ ] Sorting is consistent

**Rollback:** `git mv` back to original names
